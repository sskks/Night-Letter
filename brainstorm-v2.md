# 夜信 Night Letter — 产品脑暴记录

**主题**：MVP 之后的功能迭代方向与体验优化建议
**HMW**：我们如何能让「夜信」从一个可用的梦境记录工具，成长为一个让用户持续回来、真正帮助他们理解自己内心的情绪觉察伙伴？
**日期**：2026-06-16
**产出创意数**：22 个（筛选后保留 16 个，推荐 Top 5）

## 当前 MVP 功能清单（基线）

在发散之前，先梳理清楚我们已经有什么：

| 功能模块 | 具体能力 |
|---------|---------|
| 记录 | 文字输入 + 语音转文字（Web Speech API）、梦境/失眠模式切换、字数统计 |
| AI 对话 | Qwen 驱动 3 轮回信、本地兜底模板、"换一种说法"重新生成 |
| 情绪检测 | 关键词匹配 5 种情绪（不安/想念/困惑/平静/喜悦）+ 自动标签 |
| 可视化 | 星图（梦境星座）、情绪天气、极光粒子背景 |
| 数据分析 | 一句洞察、徽章系统（连续天数）、每周情绪报告 |
| 社交 | 梦境广场（匿名分享、点赞、留言） |
| 导出 | 梦境卡片（Canvas 渲染 PNG）+ AI 创意卡片生成 |
| 账户 | 邮箱注册/登录 + 匿名浏览、腾讯云 CloudBase 同步 |
| 提醒 | 浏览器通知（每天早上 7:00 提醒记录） |
| 引导 | 3 页 Onboarding 流程 |

## 竞品启发（关键差异点）

调研了 8 款同类/相邻产品，提炼出「夜信」的差异化机会：

| 竞品 | 亮点功能 | 夜信的差异化空间 |
|------|---------|--------------|
| Lucidity | 隐私优先、生物识别锁、搜索/筛选/收藏 | 夜信缺少搜索和隐私保护机制 |
| Somnia | 情绪/主题追踪仪表盘、AI 个性化解读 | 夜信的数据分析维度单一，仅有 5 种情绪 |
| Voneir | 5 种分析视角（心理学/星座/灵性等）、反复主题检测 | 夜信 AI 只有一种回应风格，缺少多角度解读 |
| Reflct | 每晚 3 个引导问题、情绪热力图、周报 | 夜信缺少引导式记录，用户不知写什么时会卡住 |
| Daylio | 心情日历热力图、习惯追踪、丰富统计 | 夜信缺少时间维度的可视化（月度/年度视图） |
| Oniri | 清醒梦训练、社区讨论、梦境分类标签 | 夜信的广场功能偏基础，缺少分类和深度互动 |
| Dreamy | 精美卡片分享、梦境时间线 | 夜信已有卡片导出，但缺少时间线视图 |

**核心发现**：夜信的 UI 设计语言（极光/星空/诗意文字）在同类产品中独树一帜，这是核心竞争力。但在「数据深度」「用户粘性机制」「AI 个性化」三个维度上有明显短板。

## Top 5 推荐创意

### 创意 1：AI 多人格（编号 1）

- **描述**：在 AI 对话界面增加人格选择器，提供 3 种回信风格。**温柔倾听者**（当前风格，侧重共情和接纳）、**理性分析师**（侧重认知行为分析，指出思维模式）、**诗意旅人**（侧重隐喻和意象，用散文诗的方式回应）。技术实现只需切换 system prompt。
- **解决的痛点**：AI 回应风格单一，长期使用后新鲜感消退；不同用户在不同心境下需要不同类型的回应（有时需要安慰，有时需要分析）。
- **预期效果**：预计提升对话完成率 20-30%，增加用户在对话页面的停留时间。
- **实现路径**：修改 `callQwenAPI` 的 system prompt 参数，增加 UI 人格选择器（3 个 chip），将选择存入 localStorage。
- **验证方式**：灰度发布，对比有人格选择 vs 无人格选择的对话轮数和回访率。
- **风险**：部分人格可能生成质量不稳定的回复 → 为每种人格准备充分的本地兜底模板。

### 创意 2：引导式梦境日记（编号 2）

- **描述**：在写信页面增加"不知道写什么？"入口，点击后进入引导模式。通过 4-5 个渐进式问题引导用户回忆：① 你梦到了什么地方？② 那里有谁？③ 最清晰的画面是什么？④ 醒来时是什么感觉？⑤ 如果给这个梦取个名字，你会叫它什么？每个问题独立一页，最终汇总成完整的梦境记录。
- **解决的痛点**：新手用户面对空白输入框的"空白页恐惧"；用户以为自己不记得梦了，但其实只是不知道从哪里开始回忆。
- **预期效果**：预计将"打开 app 但未记录"的流失率降低 15-25%。
- **实现路径**：新增一个引导模式的 UI 流程（5 步卡片式交互），每步收集一个短文本/选择，最后合并写入 dreamIn textarea。
- **验证方式**：统计引导模式的触发率、完成率，以及引导生成的记录 vs 自由记录的平均字数对比。
- **风险**：老用户可能觉得多余 → 仅作为可选入口，不改变原有流程。

### 创意 3：梦境搜索与标签系统（编号 5+6 合并）

- **描述**：在历史页面顶部增加搜索框（支持全文搜索），并添加标签筛选功能。标签分两层：系统标签（情绪类型）+ 用户自定义标签。同时 AI 在回信结束时自动建议 2-3 个主题标签（如 #飞行 #家人 #迷路），用户可一键添加。
- **解决的痛点**：记录积累后无法快速找到特定梦境；无法按主题维度（而不仅是情绪）分析自己的梦境模式。
- **预期效果**：预计提升历史页面的使用频率 30%+，为后续的"意象词典"和"跨梦关联"功能奠定数据基础。
- **实现路径**：localStorage 中给 dream 对象增加 `tags:[]` 字段；历史页面增加搜索 input + 标签 chip 筛选栏；`getDreams()` 支持 filter 参数。
- **验证方式**：搜索和筛选功能的使用率、标签覆盖率（有多少比例的梦境被打上标签）。
- **风险**：用户可能懒得手动加标签 → AI 自动建议 + 一键确认降低门槛。

### 创意 4：情绪日历热力图（编号 7）

- **描述**：在"发现"页面新增月历视图，每天的情绪用对应颜色的方块填充（类似 GitHub 贡献热力图）。点击某天可快速查看当天的梦境摘要。支持月度切换，底部显示本月情绪分布趋势。
- **解决的痛点**：现有的"情绪天气"只有百分比统计，缺少时间维度的直观呈现；用户难以发现情绪与日期/事件之间的关联模式。
- **预期效果**：预计成为 app 内打开频率最高的页面之一，显著提升用户对"长期记录价值"的感知。
- **实现路径**：新增 `renderCalendar()` 函数，从 localStorage 读取 dreams 数据，按日期聚合情绪，渲染 CSS Grid 日历。点击日期弹出 mini 梦境摘要卡片。
- **验证方式**：日历页面访问量、月份切换操作次数。
- **风险**：数据量极少时日历显得很空 → 设计优雅的空状态（渐变底色 + 鼓励性文案）。

### 创意 5：解除对话限制 + 每日微签到（编号 3+15 合并）

- **描述**：
  - **对话轮数**：将 3 轮硬性限制改为 5 轮默认 + 设置中可选"无限对话"。结束时不再是硬性关闭，而是 AI 自然地道别，用户仍可继续说最后一句。
  - **微签到**：首页增加一个快速情绪签到入口（5 个情绪表情图标），点击即完成当日签到，不需要写任何东西。签到数据也计入连续天数和情绪统计。
- **解决的痛点**：对话被强制结束的体验生硬且令人沮丧；没做梦或没时间写的日子里用户完全没有互动，导致连续天数断裂→失去动力。
- **预期效果**：对话满意度提升（不再有"话没说完"的挫败感）；日活提升 15-20%（微签到降低了参与门槛）。
- **实现路径**：修改 `maxTurns` 变量和 `finishChat()` 逻辑；首页新增签到 UI（5 个表情按钮 + 签到动画），签到数据存入 `nightletter_checkins` localStorage key。
- **验证方式**：平均对话轮数变化、微签到功能的日使用率、连续天数的平均值变化。
- **风险**：无限对话可能增加 API 成本 → 设一个软上限（如 15 轮后 AI 温柔收尾）。

## 被放弃的创意（存档参考）

- **梦境图片上传（#17）**：放弃原因 — 单文件 HTML 架构下图片处理复杂度高，且需要 CloudBase 文件存储，开发成本远大于收益。
- **亮色主题（#21）**：放弃原因 — 夜信的品牌定位就是"夜晚的、深沉的、星空感的"，亮色主题与产品气质冲突。
- **多语言（#22）**：放弃原因 — 当前用户群为中文用户，国际化优先级低。

## 附：现有 Bug 修复建议

在实现新功能之前，建议先修复以下已知问题：

1. **注册邮箱格式无校验** — `doRegister()` 缺少邮箱正则校验，用户可输入任意字符串注册。（已修复）
2. **注册后广场提示未登录** — CloudBase `signUpWithEmailAndPassword` 不自动登录，`_getCurrentUser()` 返回 null。需要在注册后显式调用 `signInWithEmailAndPassword`。（已修复）
3. **`_getCurrentUser()` 同步调用异步 API** — `cbAuth.getLoginState()` 返回 Promise，但 `_getCurrentUser()` 同步返回，导致页面刷新后登录态丢失。已改为缓存模式 + `onLoginStateChanged` 监听。（已修复）
4. **广场分享使用 `prompt()` / `confirm()`** — 原生弹窗在移动端体验差，建议替换为 app 内自定义弹窗（复用 `confirmModal` 组件）。
5. **梦境广场 `loadSquareDreams` 排序字段** — 代码中 `orderBy('createTime','desc')` 但 `add` 时没有写入 `createTime` 字段，可能导致排序失效。应在 add 时加上 `createTime: new Date().toISOString()`。
